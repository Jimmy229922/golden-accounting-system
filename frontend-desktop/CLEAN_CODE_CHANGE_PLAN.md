# خطة تنظيف الكود وتنظيمه (Clean Code Plan)

## الملخص
- **النطاق:** `frontend-desktop` فقط
- **الهدف:** رفع الجودة من **45%** إلى مستوى تشغيلي وصياني مستقر بدون كسر السلوك الحالي
- **المنهج:** تحسين تدريجي مرحلي `P0 -> P3` مع تثبيت baseline وقياس تقدّم رقمي
- **قاعدة التنفيذ:** لا تغيير في مخرجات المستخدم النهائية إلا بعد تحقق التكافؤ الوظيفي

## تغييرات APIs / Interfaces / Types
- **في مهمة التوثيق الحالية:** لا توجد أي تغييرات على APIs أو Types أو سلوك التطبيق.
- **في خطة التحسين المستقبلية:** التغييرات المقترحة داخلية (Internal Refactoring Boundaries) مع الحفاظ على عقود `window.electronAPI` وسلوك الصفحات.

## خطة التنفيذ حسب الأولوية

### P0 - تثبيت الجودة قبل أي تفكيك
| العنصر | التفاصيل |

## أهم التغييرات المطلوب تنفيذها (قابلة للتنفيذ مباشرة)
1. **توحيد Navigation**
- جعل `navigation.json + navManager.js` المصدر الوحيد للتنقل.
- إزالة كل `getNavHTML` مكرر تدريجيًا مع فحص تكافؤ السلوك.

2. **استخراج Helpers مشتركة**
- نقل `t/fmt` إلى shared module واحد.
- توحيد bootstrap pattern للصفحات بدل تكرار `DOMContentLoaded` بطرق مختلفة.

3. **تقليل Inline Handlers**
- استبدال `onclick/onchange/oninput` داخل templates بـ delegated listeners مرتبطة بعناصر root.
- الإبقاء على API handlers بنفس التواقيع منعًا لكسر السلوك.

4. **تفكيك الملفات الضخمة**
- اعتماد تقسيم ثابت داخل كل صفحة:
  - `state`
  - `services/api`
  - `renderers`
  - `events/actions`
  - `page-bootstrap`

5. **إضافة Quality Gates**
- lint + smoke checks كحد أدنى.
- منع الدمج إذا زادت مؤشرات التدهور عن baseline.

## سيناريوهات الاختبار والقبول
1. تحميل كل صفحة رئيسية بدون أخطاء Console blocker.
2. تنفيذ رحلة إصدار فاتورة بيع كاملة (إضافة/تعديل/حذف) مع مطابقة النتائج قبل/بعد.
3. تنفيذ رحلة مشتريات كاملة مع مطابقة الرصيد والمخزون.
4. تنفيذ رحلة مردودات بيع/شراء مع مطابقة الحركات المالية.
5. اختبار صلاحيات العرض والإجراءات (admin/non-admin).
6. اختبار الترجمة والتنقل عبر جميع الصفحات بعد توحيد الـnav.

## Definition of Done (مؤشرات نجاح قابلة للقياس)
- خفض ملفات JS أكبر من 300 سطر من **27** إلى **<= 12**.
- خفض ملفات JS أكبر من 500 سطر من **14** إلى **<= 4**.
- خفض تكرار `getNavHTML` من **16** إلى **0**.
- خفض تكرار `t/fmt` من **25** إلى **<= 2** (مصدر موحد).
- خفض inline handlers من **63** إلى **<= 10**.
- خفض globals المعروضة على `window` من **29** إلى **<= 8**.
- اعتماد فعلي لـ `navManager` في جميع صفحات الواجهات الأساسية.
- وجود lint checks وتشغيلها بشكل ثابت قبل الدمج.
- وجود smoke suite تغطي الرحلات الحرجة (Sales/Purchases/Returns/Reports/Auth).

## افتراضات التنفيذ
- لا تعديل على قاعدة البيانات أو behavior الوظيفي إلا عند الحاجة وبموافقة واضحة.
- التنفيذ سيكون تدريجيًا مع مقارنة قبل/بعد لكل مرحلة.
- أي dependency جديدة أو تعديل build scripts يتطلب موافقة مسبقة أثناء التنفيذ الفعلي.
